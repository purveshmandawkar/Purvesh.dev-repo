Autonomous Thermal Monitor & Alert System

A standalone, battery-powered thermal safety device for industrial environments. Runs entirely on bare-metal ATmega firmware — no Arduino library dependency — with a three-tier alert architecture and cumulative alarm logic.

Bare-metal Firmware Analog Sensing Battery Optimization Standalone Embedded ATmega8A

Three-Tier Alert Architecture

Unlike a simple ON/OFF alarm, this system uses cumulative alerting — the warning state stays active even when the critical state is triggered, giving operators a visual history of thermal escalation.

🟢
Zone 1 — Normal
< 60°C

Green LED only. System healthy.

🟡
Zone 2 — Warning
60–70°C

Yellow LED. No buzzer yet.

🔴
Zone 3 — Critical
> 70°C

Yellow latched + Red pulsing + Buzzer.

The yellow LED latches in the critical zone — it stays ON even as the red alert triggers. This tells the operator: the system passed through the warning zone before reaching critical. Standard alarms don't preserve this history.

Going Standalone — The Key Challenge

The original prototype ran on an Arduino Uno. To transition to a portable standalone device, two hardware changes were required:

1. Internal clock: The ATmega8A was reconfigured to use its internal 8MHz RC oscillator instead of an external crystal. This eliminates a mechanical component that can fail and reduces current draw for battery life.

2. Noise suppression: When the buzzer triggered, its inductive load caused voltage sags that reset the microcontroller. This was fixed with a 0.1µF decoupling capacitor directly across the IC power pins and a star-grounding layout — all ground connections meet at a single point to prevent ground loops.

Source Code

Thermal Monitor Firmware — Cumulative alert logic

const int tempPin   = A0;  // LM35 analog input (Physical Pin 23)
const int greenLed  = 8;   // Normal state (Physical Pin 14)
const int yellowLed = 2;   // Warning state (Physical Pin 4)
const int alertPin  = 0;   // Critical alert + buzzer (Physical Pin 2)

const int warnLimit  = 60; // °C — Yellow activates
const int alertLimit = 70; // °C — Red + Buzzer activates

void setup() {
  pinMode(greenLed,  OUTPUT);
  pinMode(yellowLed, OUTPUT);
  pinMode(alertPin,  OUTPUT);
}

void loop() {
  int reading = analogRead(tempPin);
  // LM35: 10mV/°C, ADC reference 5V, 10-bit resolution
  // temperature = (reading * 500) / 1023

  if (reading >= alertLimit) {
    // ZONE 3: CRITICAL — yellow latches, red pulses with buzzer
    digitalWrite(greenLed,  LOW);
    digitalWrite(yellowLed, HIGH);  // Latched: shows warning was passed

    // Pulsed alert — 100ms ON / 100ms OFF
    digitalWrite(alertPin, HIGH);
    delay(100);
    digitalWrite(alertPin, LOW);
    delay(100);

  } else if (reading >= warnLimit) {
    // ZONE 2: WARNING — yellow only, no audible alarm yet
    digitalWrite(greenLed,  LOW);
    digitalWrite(yellowLed, HIGH);
    digitalWrite(alertPin,  LOW);

  } else {
    // ZONE 1: NORMAL — green only, all alerts off
    digitalWrite(greenLed,  HIGH);
    digitalWrite(yellowLed, LOW);
    digitalWrite(alertPin,  LOW);
  }
}

Engineering Challenges Solved

ProblemSolution
Buzzer causing MCU resets0.1µF decoupling capacitor across VCC/GND pins absorbs voltage sags from inductive buzzer load
Ground loops causing noiseStar-grounding technique — all grounds meet at single point, preventing noise coupling between circuits
Arduino dependencyReconfigured ATmega8A to use internal oscillator — runs as fully standalone chip, no bootloader required
Simple ON/OFF alarms lose contextCumulative alerting latches yellow LED in critical zone — operator knows the warning stage was crossed

Key Insight

"Transitioning from a prototype to a standalone device taught me that noise is a real hardware problem, not just theory. A buzzer resetting a microcontroller is fixed with a capacitor, not more code."
